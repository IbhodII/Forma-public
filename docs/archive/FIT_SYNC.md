# Импорт FIT-файлов (Coospo Ride)

Пошаговая инструкция для пользователя и справка для разработчика: куда класть файлы, как запустить импорт, что происходит с уже существующими тренировками и зачем нужен `--reimport`.

Связанные документы: [BIKE.md](./BIKE.md) (карта, графики, API точек), [API.md](./API.md) (`/api/sync/fit`), корневой [SYNC.md](../SYNC.md) (все скрипты синхронизации).

---

## Кратко

| Вопрос | Ответ |
|--------|--------|
| Какие файлы? | Только **`.fit`** / **`.FIT`** (велокомпьютер Coospo Ride и совместимые записи) |
| Куда класть? | Папка из **Настройки → Синхронизация** или `E:\fit activity` / `./fit_files` |
| Как запустить в UI? | **Настройки** → вкладка **Синхронизация** → **Импорт FIT** |
| Повторный импорт того же файла? | По умолчанию **пропуск**; полное обновление — **`--reimport`** (CLI) или API с `reimport: true` |
| Ключ тренировки в БД | **`start_time`** (локальное время старта из FIT) |
| Polar | Attach из очереди pending — см. [SYNC.md](../SYNC.md) |

## 1. Где лежат FIT-файлы

### Порядок выбора папки

Импортёр (`fit_importer.py` / API) выбирает каталог так:

1. **Явный путь** — аргумент CLI `--folder`, тело API `{ "folder": "..." }` или поле в запросе «Все интеграции».
2. **Настройки приложения** — `user_profile.fit_folder_path` (сохраняется на вкладке **Синхронизация**).
3. **По умолчанию на диске** — `E:\fit activity`, если эта папка существует.
4. **Запасной вариант** — `./fit_files` в корне проекта.

Подпапки **сканируются рекурсивно** (`rglob`): можно хранить файлы в подкаталогах по годам или поездкам.

### Как изменить путь в интерфейсе

1. Откройте дашборд → **Настройки** (`/settings`).
2. Вкладка **Синхронизация** (`/settings?tab=sync_cloud`).
3. Поле **«Папка с FIT-файлами»** — абсолютный путь (`D:\Ride\Coospo`) или относительный (`fit_files/`).
4. Нажмите **Сохранить**.
5. Под полем показывается **«Сейчас импорт использует: …»** (`effective_fit_folder_path`) — итоговый каталог с учётом дефолтов.

Пустое поле после сохранения означает: использовать цепочку п. 3–4 (не хранить свой путь в профиле).

### Типичный рабочий процесс

1. Подключите велокомпьютер (Coospo Ride) к ПК.
2. Скопируйте новые `.fit` в выбранную папку (или оставьте экспорт сразу в `E:\fit activity`).
3. Запустите импорт из UI или CLI (см. ниже).
4. Обновите страницу **Тренировки → Велосипед** — появятся заезды с картой и графиками.

## 2. Запуск импорта из интерфейса

### Кнопка «Импорт FIT»

**Путь:** Настройки → Синхронизация → блок **«Синхронизация данных»** → **Импорт FIT**.

Что происходит:

1. Отправляется `POST /api/sync/fit` с папкой из настроек (или из поля формы, если передано в компонент).
2. Сервер запускает импорт **в фоне** и сразу отвечает `{ "status": "started", "task_id": "..." }`.
3. Интерфейс показывает прогресс (обработано файлов / всего) и опрашивает `GET /api/sync/fit/status/{task_id}` каждые ~2,5 с.
4. По завершении — toast: сколько файлов добавлено, обновлено, пропущено, ошибок.

Пока импорт идёт, повторный запуск блокируется (**409** — уже выполняется другая задача). Кнопка **«Все интеграции»** в это время недоступна.

### Кнопка «Все интеграции»

Запускает `POST /api/sync/integrations`: **FIT** (синхронно, запрос ждёт окончания) и **Polar fetch** (очередь pending). Для больших папок FIT удобнее отдельная кнопка **Импорт FIT** (фон).

### Переимпорт из UI

Кнопка **Импорт FIT** по умолчанию работает как **обычный** импорт (без `reimport`): уже учтённые файлы пропускаются. Полный переимпорт всех `.fit` из UI отдельным переключателем не вынесен — используйте CLI или API (раздел 5).

### Ошибки в UI

| Ситуация | Что увидите |
|----------|-------------|
| Папка не найдена / нет доступа | Сообщение с просьбой проверить путь в Интеграциях |
| Нет `.fit` в папке | «Не найдено FIT-файлов в папке…» |
| Импорт уже идёт | Ошибка 409, дождитесь завершения |

## 4. Обычный импорт: новые файлы и пропуски

### Журнал файлов `imported_files`

Для каждого обработанного `.fit` сохраняется запись:

- `file_name` — имя файла (например `2025-05-20-12-30-45.fit`);
- `source` — `fit_coospo`;
- `imported_at` — время импорта.

При **обычном** запуске (без `--reimport`):

- если файл **уже в** `imported_files` **и** тренировка в БД на месте → файл **пропускается** (`skipped`);
- если файл помечен импортированным, но строки в `cardio_workouts` нет → **repair**: повторная загрузка (`repaired`).

### Upsert тренировки по `start_time`

Из FIT читается время старта сессии → строка **`start_time`** (ISO, локальное время без микросекунд).

В `cardio_workouts` (тип **`вело`**):

- если есть строка с тем же **`type` + `start_time`** → **UPDATE** (дистанция, длительность, пульс, каденс, ккал, `data_source = fit_coospo`, …);
- иначе → **INSERT** новой тренировки.

Несколько заездов **в один календарный день** — отдельные строки с разным `start_time` (ограничение «одна вело-строка на дату» снято для FIT).

После upsert для этой тренировки:

- перезаписываются **`workout_heart_rate`**, **`workout_sensors`**, **`gps_tracks`** (старые точки удаляются, вставляются заново из файла);
- файл помечается в **`imported_files`**.

### Что попадает из FIT

| Данные | Таблица / поле |
|--------|----------------|
| Сводка заезда | `cardio_workouts` |
| Пульс по секундам | `workout_heart_rate` |
| Скорость, каденс, высота, температура, мощность | `workout_sensors` |
| Маршрут | `gps_tracks.track_data` (GeoJSON) |

Парсер: **`fitdecode`** (`parse_fit_file` в `fit_importer.py`). GPX/TCX **не** поддерживаются.

### Статистика в конце

В логе/ответе API:

- **imported** — новые успешные импорты;
- **repaired** — восстановление после «битой» отметки в `imported_files`;
- **skipped** — пропуск (уже импортировано или дата в будущем);
- **errors** — ошибка разбора файла.

## 5. Флаг `--reimport` (полный переимпорт)

### Зачем

- обновить GPS / датчики после изменения логики импортёра;
- исправить повреждённую тренировку;
- заново прочитать файлы, которые раньше были пропущены как «уже импортированные».

### CLI

```powershell
.\venv\Scripts\python.exe fit_importer.py --reimport
.\venv\Scripts\python.exe fit_importer.py --folder "E:\fit activity" --reimport
```

### API

```http
POST /api/sync/fit
Content-Type: application/json

{
  "folder": "E:\\fit activity",
  "reimport": true
}
```

Синхронный режим (отладка): `POST /api/sync/fit?sync=true` с тем же телом.

### Отличие от обычного импорта

| | Обычный | `--reimport` |
|---|---------|----------------|
| Проверка `imported_files` | Пропуск известных файлов | **Не** пропускает по журналу |
| Перед записью | — | Удаляет существующую тренировку с тем же `start_time` (и связанный HR/GPS) |
| Запись в `imported_files` | Да | Да (обновляет отметку) |
После `--reimport` обновите страницу велотренировок в браузере (F5).

## 6. Для разработчика

### Цепочка вызовов

```
UI SyncButton / CLI
  → run_import(folder_path, reimport=…)
    → get_fit_folder_path()
    → import_fit_folder()
      → list_fit_files() / parse_fit_file() / save_workout_to_db()
      → mark_file_imported()
      → migrate_fit_calories_watch_to_chest()  (при новых/repair/reimport)
```

Фон API: `backend/services/fit_importer_service.py` → поток → `run_import(..., on_progress=…)`.

### Ключевые модули

| Файл | Роль |
|------|------|
| `fit_importer.py` | Парсинг FIT, CLI, `run_import` |
| `database/db_utils.py` | `upsert_cardio_workout`, миграция ккал FIT |
| `backend/routers/sync.py` | `POST /api/sync/fit`, статус задачи |
| `backend/services/fit_importer_service.py` | Фоновые задачи, 409 при параллельном импорте |
| `utils/bike_track.py` | GeoJSON для карты |

### API (кратко)

| Метод | Путь | Назначение |
|--------|------|------------|
| POST | `/api/sync/fit` | Старт фонового импорта (по умолчанию) |
| GET | `/api/sync/fit/status/{task_id}` | Прогресс и результат |
| POST | `/api/sync/fit?sync=true` | Синхронный импорт (блокирует до конца) |
| POST | `/api/sync/integrations` | FIT + Polar fetch |

Тело `FitSyncBody`: `folder`, `folder_path` (alias), `reimport`.

### Настройки пользователя

- GET/POST `/api/user/integration-settings` — `fit_folder_path`, `effective_fit_folder_path`.
- Хранение: `user_profile.fit_folder_path` в `workouts.db`.

### Очистка дубликатов

После обхода файлов: **`cleanup_stale_fit_bike_duplicates()`** — убирает устаревшие дубли FIT-вело, оставляя актуальные записи по `start_time`.

## 8. Частые вопросы

**Импорт прошёл, но заезда нет в списке**  
Проверьте вкладку **Велосипед**, фильтр даты и что файл не с датой в будущем (такие файлы пропускаются).

**Карта пустая, а FIT импортирован**  
Откройте тренировку → вкладка карты; при необходимости `--reimport` для пересборки `gps_tracks`.

**Хочу только новые файлы**  
Обычный импорт без `--reimport` — достаточно.

**Поменял папку в настройках, старые файлы не подхватились**  
Импорт читает **новую** папку; файлы нужно скопировать туда или указать путь, где они лежат.
